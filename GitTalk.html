About the talk

What is git?

  • Git is a distributed version control system
  • Allows for multiple people to work on the same project for group work
  • Allows you to commit changes to code so you can easily revert back
  • Allows you to create features but not break existing code, through branches

Git vs Github

git is the system Github is just a host. There are other hosts such as BitBucket and Stash.

How git works

  • Keeps track of incremental changes to files and a their history
  • Each repository is a mirror of the server
  • If the server goes down it can be restored from any clone

Why use git vs other VCS

  • Git is distributed
  • This means that you can work offline
  • Git init will make a local repo that can be used locally and does not need to be hosted, useful for homework.
  • You have a back up of your work online if you host it. (In case someone spills coffee on your computer)

How do I git?

Setting up a repo

Two different ways to create a repo on your machine:

Preexisting Directories

 git init
Later a remote repository can be added with. 
 git remote add origin https://github.com/user/repo.git

Cloning

If you have a preexisting repository you can clone it with 
git clone [url]
This will create a folder with the repo name and copy the git repository.

Git Commands

Git Status

Checks the status of the repo.
Displays the current branch, and how it compares to the online version.

git status
Results divided into 3 sections:
Changes to be committed these changes will be committed when git commit is used.
Changes not staged for commit these are changes to files already registered with the repo, but need to be git added.
Untracked files these are changes on files that are not yet tracked by the repo, they will also need to be added.

Git Add

In order for git to track a file you must tell git to add it. This is done by

git add [filename]

Ensure that you want to track the file, compiled code, and extraneous files should be added to the .gitignore

Other uses of git add include git add -A which will add all files

When you start to get comfortable with git start tyring to use the wildcard * to add multiple files at once. The -p is also incredibly powerful but not for the faint of heart.

Git Commit

Once you have added a file you must make a commit. Commits are basically groups of changes a developer makes and have a message describing what they are. It is good practice to make your commits small and logical. This can help you in the future.

To create a commit use:

git commit -m "[message]"

If you use git commit -am "[message]"All modified tracked files will be added.

Git Push

Once you have made your commits and are ready to share them with the world you need to push. Note: There may be issues if someone has pushed before you so be sure to follow the instructions provided, usually a git pull.

When using command line I usually find it best to only push the branch I am working on, to do so use

git push origin [branch name]

Git push may prompt you for your username and password so be sure to provide the correct pair.

Once pushed you can now go to your host and see your changes.

Git Pull

Once your code is on the repo the other contributors can git pull your changes and make their own edits. I tend to do one branch at a time.

git pull origin [branchName]

Git pull may produce merge errors which are covered later.

If the pull is successful then you will see a diff log of all the files changed and the number or additions or deletions.

Git Branch

Branching is very important in git. You usually want to keep your master stable so branches allow for the developer to add features with out breaking everyone else's work or are easily able to go back to where they were before.

To create a new branch use:

git branch [branchName]

Once created you will need to use git checkout to switch to the branch.

The branch command has other uses as well.

git branch

Will list all branches that your local machine is aware of, you are able to checkout branches not on this list.

Once a branch is merged into master and you are done with it, it is good practice to delete it with git branch -d [branchName].

Git Checkout

In order to start using your branch run:

git checkout [branchName]

Git checkout and git branch can be combined by using git checkout -b [branchName]. This will make and checkout the branch.

Git Merge

When a branch is ready for production or to be combined with another branch you run:

git merge [sourceBranchName]

This command tells git to merge the two branches and their files together and determine what should be kept and what discarded.

Merge conflicts may occur when doing this and need to be resolved before the merge commit is ready.

Git Stash

A useful is not abused command is

git stash

This will put your changes in a stack. The changes can either be ignored or brought back later.

Git Diff

In order to look at the changes that have been made to unstaged changes you can use

git diff

This will show you a list of additions and deletions that have been made.q to exit.

Git Rm

To remove files from the index use git rm [filename]

Git Mv

To rename files or move them use git mv [source] [destination].

Merging

If your commits are structured wrong you may end up with merge conflicts. Multiple developers changing the same line of code may cause this, so if possible try to work in different areas.

Merges may be scary but there are tools to help. My personal favorite is Kdiff3

KDiff will allow you to go through each conflict and either choose the change you want or edit the changes until you are happy. Once complete save and exit, git will now be able to finish its merge.

If you want to setup Kdiff3 as your default merge tool run the following commands to modify your .gitconfig

git config --global merge.tool kdiff3
git config --global mergetool.kdiff3.keepBackup false
git config --global mergetool.kdiff3.path "C:\Program Files\KDiff3\kdiff3.exe"

To use the merge tool use git mergetool then use the UI to resolve any issues.
